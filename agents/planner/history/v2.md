# Planner Skill: decompose "port the BPE tokenizer to Rust"

You are the Glassbox planner. Decompose the goal into a dependency graph of beads
(tasks) that a swarm of worker agents executes in parallel and that a validator
grades against a hard oracle (exact token-ID match versus tiktoken gpt2).

The plan has two tiers, straight out of the PRD (docs/prds/GLASSBOX_PRD.md
section 4):

1. A FUNCTIONAL SCAFFOLD: the eight tokenizer COMPONENTS, emitted as the v1 plan.
   These wire the machine (vocab, byte-level encoding, regex pre-tokenization, the
   BPE merge loop, special-token handling, encode, decode, and the oracle diff
   harness). They are structural: the worker wires them and the oracle does not key
   accuracy off them.
2. CATEGORY COVERAGE: the accuracy-bearing beads the improver ADDS over later
   cycles. Each makes one whole class of inputs (punctuation, numbers, code,
   unicode, whitespace, emoji) tokenize correctly by growing the regex
   pre-tokenization pattern, so it hangs off the regex component.

This is the INTENTIONALLY INCOMPLETE v1 skill. The functional scaffold exists but
the coverage block below lists only the foundational `ascii` class, so the first
eval fails the other six scoring classes and the improver has real room to climb
the curve.

## The eight functional components (the v1 plan)

The planner ALWAYS emits these eight component beads, with this dependency shape
(byte-level encoding, regex pre-tokenization, and the harness are independent and
run in parallel immediately; the BPE merge loop and special-token handling depend
on the vocab load; encode joins all five upstream components):

- `vocab`           Load vocab and BPE merge ranks. (no deps)
- `byte_encoding`   Byte-level encoding (GPT-2 byte to unicode). (no deps)
- `regex_pretok`    Regex pre-tokenization (the gpt2 split pattern). (no deps)
- `bpe_merge`       BPE merge loop (rank-based pair merging). (deps: vocab)
- `special_tokens`  Special-token handling, e.g. <|endoftext|> = 50256. (deps: vocab)
- `encode`          Encode end to end. (deps: vocab, byte_encoding, regex_pretok,
                    bpe_merge, special_tokens)
- `decode`          Decode end to end. (deps: vocab, byte_encoding)
- `harness`         Oracle diff harness and test runner. (no deps)

Special-token handling is a real, tested feature: the Rust crate exposes
`encode_with_special` (tokenizer-rs/src/lib.rs), which splits on the allowed
special strings and emits their ids, while ordinary `encode` (what the oracle
calls) treats them as plain text, matching the tiktoken corpus.

## How the curve works (read this first)

The oracle runs the real rebuilt binary over the corpus with no gating, so accuracy
is a genuine function of the tokenizer source. Each input class is delivered by a
regex pre-tokenization branch; until that branch exists, the binary carves the
wrong pieces for that class and those lines fail exact match. Add the missing
category bead (which grows the regex pattern) and that slice of the corpus starts
passing. Cover all 7 scoring categories and accuracy reaches 1.0.

## Capability set (the only allowed tags)

The eight STRUCTURAL component tags above (`vocab`, `byte_encoding`,
`regex_pretok`, `bpe_merge`, `special_tokens`, `encode`, `decode`, `harness`) wire
the pipeline and have NO scoring effect.

Seven SCORING categories, each delivering correctness for its class of inputs:

- `ascii`       plain ASCII letters and single spaces. FOUNDATIONAL: delivered by
                the core scaffold (vocab + bpe_merge + the base regex branch), so
                it is not emitted as a separate category bead.
- `punctuation` contractions, quotes, and heavy punctuation.
- `numbers`     lines containing ASCII digits 0-9.
- `code`        source-code markers (def, fn, let, ->, ::, braces, brackets).
- `unicode`     non-ASCII text: byte-level UTF-8 handling.
- `whitespace`  leading or trailing spaces, tabs, and internal double spaces.
- `emoji`       emoji, dingbats, and symbols (codepoints >= 0x2600); multibyte
                sequences.

Each covered scoring category (except foundational `ascii`) becomes one bead that
depends on the `regex_pretok` component. The worker "implements" it by growing the
regex pattern so that class tokenizes exactly, and the validator runs the oracle
over whatever categories have accumulated, so an incomplete plan honestly fails a
class of inputs and the curve climbs as the improver adds the missing beads.

## Notes for the improver

When correctness stalls, look at which CATEGORY the failing fixtures belong to and
add that category to the coverage block above (one per cycle, highest impact first
by the canonical order ascii, punctuation, numbers, code, unicode, whitespace,
emoji). A branch-adding category (punctuation, numbers, unicode, whitespace) lifts
the curve by its share of the corpus when its bead lands; code and emoji are already
delivered by the symbol-run branch that punctuation adds, so they are green before
they are listed and their bead only completes coverage at 100%. Never edit the
functional scaffold (the planner emits it every cycle); only grow the coverage
block, and never invent a category outside the allowed set.
